# AI 機能について（ndlocr-lite 矩形選択半自動化と OCR）

## 1. 機能概要

| 機能                     | 説明                                                                                                                                                                 |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **半自動矩形選択**       | 「AIで自動矩形選択」ボタンで、現在ページの画像を AI で解析し、**ブロック単位**（見出し・段落・表・図など）の矩形を検出して、読み順で選択状態にする。                 |
| **OCR テキストレイヤー** | ページを表示すると、バックグラウンドでそのページの OCR を実行し、**行単位**で認識したテキストを PDF の上に透明レイヤーとして重ねて表示する。検索・選択しやすくなる。 |

どちらも **[NDL OCR（国立国会図書館の OCR パイプライン）](https://github.com/ndl-lab/ndlocr-lite)** のロジックをもとにして、さらに **ONNX Runtime Web（WASM）** を使って**ブラウザ上で動かしています。**

## 2. OCR の全体パイプライン（画像からテキストまで）

```
  [PDFページ画像]
        │
        ▼
  ┌─────────────────────────────────────────────────────────────┐
  │ 1. 前処理（画像の正規化）                                      │
  │    キャンバスから画像データ取得・必要に応じたリサイズ・パディング   │
  └───────────────────────────┬─────────────────────────────────┘
                              │
                              ▼
  ┌─────────────────────────────────────────────────────────────┐
  │ 2. レイアウト検出（どこに何があるか）                           │
  │    「テキストブロック」「行」「表・図・広告」などの領域を検出       │
  │    → このプロジェクトでは DEIM モデルが担当                      │
  └───────────────────────────┬─────────────────────────────────┘
                              │
              ┌───────────────┴───────────────┐
              ▼                               ▼
  ┌─────────────────────────┐     ┌─────────────────────────────┐
  │ 3a. ブロック単位の矩形   │     │ 3b. 行単位の切り出し         │
  │ テキストブロック・表・図  │     │ 検出された「行」を読み順に並べる │
  │ をまとめて矩形化         │     │ （XY-cut による読み順整序）    │
  │ → 半自動矩形選択で使用   │     │ → OCR テキストレイヤーで使用   │
  └─────────────────────────┘     └───────────────┬─────────────┘
                                                  │
                                                  ▼
                                    ┌─────────────────────────────┐
                                    │ 4. 文字認識（行画像 → 文字列） │
                                    │ 各行の画像を 1 本の文字列に変換  │
                                    │ → このプロジェクトでは PARSEQ  │
                                    └───────────────┬─────────────┘
                                                    │
                                                    ▼
                                    [ 行ごとの bbox + テキスト ]
                                    → 画面上の透明レイヤーで表示
```

### パイプラインの要点

| 段階                                     | 役割                                               | このプロジェクトでの担当                     |
| ---------------------------------------- | -------------------------------------------------- | -------------------------------------------- |
| **前処理**                               | 画像をモデルが扱いやすい形にする                   | キャンバスから画像取得・リサイズ・パディング |
| **レイアウト検出（セグメンテーション）** | 「どこにテキスト／表／図があるか」を領域として出す | **DEIM**（ONNX モデル）                      |
| **ブロック矩形・読み順**                 | 検出領域をブロックにまとめ、読み順を付ける         | テキストブロックのグルーピング + **XY-cut**  |
| **文字認識**                             | 行画像 → 文字列                                    | **PARSEQ**（ONNX モデル）                    |

- **半自動矩形選択**: 1 → 2 → 3a まで（ブロック矩形 + 読み順で選択状態にする）。
- **OCR テキストレイヤー**: 1 → 2 → 3b → 4（行単位で認識し、画面上にテキストを重ねる）。

## 3. 技術スタックと役割

| 技術要素                    | 役割                                                                                                   | 補足                                                                                   |
| --------------------------- | ------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------- |
| **DEIM（ONNX）**            | レイアウト検出。画像から「テキストブロック」「行」「表・図・広告」などの領域（矩形・ポリゴン）を出す。 | 前処理・後処理はプロジェクト内のレイアウト検出用モジュールが担当。                     |
| **PARSEQ（ONNX）**          | 文字認識。行画像 1 枚を入力に、1 本の文字列を出力。                                                    | 文字セット（charlist）は NDL 用設定を参照。                                            |
| **読み順整序（XY-cut）**    | 検出されたブロックや行に「上→下・左→右」の読み順（ランク）を付ける。                                   | NDL OCR と同様の考え方で実装。                                                         |
| **ONNX Runtime Web**        | ブラウザ上で DEIM / PARSEQ の ONNX モデルを実行するランタイム。                                        | WASM 版を使用。マルチスレッド時は SharedArrayBuffer と Cross-Origin Isolation が必要。 |
| **IndexedDB（キャッシュ）** | レイアウト検出結果と OCR 結果をページ単位で保存。再表示時に再計算を減らす。                            | レイアウト用と OCR 用の 2 種類のキャッシュを利用。                                     |
| **Web Worker**              | 重い計算（DEIM / PARSEQ）をメインスレッドから切り離して実行。                                          | キューで「同時に 1 タスク」に制限し、表示中のページを優先。                            |

## 4. 全体の構成

実装は、次の 3 つのレイヤーに分けています。

```
┌─────────────────────────────────────────────────────────────────┐
│ ① UI レイヤー（メインスレッド）                                    │
│    PDF 表示・ツールバー・矩形オーバーレイ・OCR テキストレイヤー表示   │
│    ユーザー操作の受け付けと、Worker へのタスク依頼・結果の反映        │
└───────────────────────────┬─────────────────────────────────────┘
                            │ 画像データの送信 / 検出結果・OCR結果の受信
                            ▼
┌─────────────────────────────────────────────────────────────────┐
│ ② 処理レイヤー（Web Worker）                                       │
│    ・レイアウト検出: 画像 → DEIM → 領域一覧（detections）            │
│    ・ブロック矩形: 領域 → テキストブロック/ブロックにまとめる → 読み順 → 矩形一覧 │
│    ・OCR: 領域から行を抽出 → 読み順 → 行ごとに PARSEQ → 行テキスト一覧 │
│    キャッシュがあれば「レイアウト検出」をスキップし、矩形計算や PARSEQ だけ実行 │
└───────────────────────────┬─────────────────────────────────────┘
                            │ 結果の保存・読み出し
                            ▼
┌─────────────────────────────────────────────────────────────────┐
│ ③ キャッシュレイヤー（IndexedDB）                                   │
│    ・レイアウトキャッシュ: ページごとの「レイアウト検出結果」+ 画像サイズ  │
│    ・OCR キャッシュ: ページごとの「行の位置 + テキスト」+ 画像サイズ    │
│    半自動矩形選択・OCR の両方でレイアウト結果を再利用可能               │
└─────────────────────────────────────────────────────────────────┘
```

- **メインスレッド**: 表示と UI、タスクの投入・結果の受け取り、キャッシュの読み書き。
- **Worker**: 画像またはキャッシュ済みレイアウトを受け取り、必要なだけ「ブロック矩形」または「行＋文字認識」を計算して返す。
- **キャッシュ**: 同じページを再度開いたときに、レイアウト検出や OCR の再実行を減らす。

## 5. 「単位」の違い（半自動矩形選択 vs OCR テキストレイヤー）

どちらも同じ「レイアウト検出結果」を土台にしていますが、**扱う単位**が違います。

| 機能                     | 使う単位       | 内容                                                                                                           |
| ------------------------ | -------------- | -------------------------------------------------------------------------------------------------------------- |
| **半自動矩形選択**       | **ブロック**   | 1 つのテキストブロック（段落など）＝ 1 矩形、1 つのブロック（表・図・広告など）＝ 1 矩形。行の矩形は使わない。 |
| **OCR テキストレイヤー** | **行（line）** | レイアウト検出で得た「行」を読み順に並べ、行ごとに文字認識。表示は「行の位置（bbox）+ テキスト」。             |

- 矩形選択: **ブロック（段落・表・図など）** の矩形のみ。
- OCR レイヤー: **行** の矩形と、その行のテキスト。

## 6. 利用シーン別の流れ（抽象化したステップ）

### 6.1 AIで自動矩形選択

1. ユーザーが「AIで自動矩形選択」を押す。
2. そのページの画像データを Worker に渡すか、すでにレイアウト検出結果がキャッシュにあればそれを渡す。
3. Worker 側で
   - キャッシュがない場合: レイアウト検出（DEIM）を実行 → 領域一覧を得る。
   - キャッシュがある場合: 領域一覧は使わず、キャッシュの検出結果を使う。  
     いずれも「領域 → テキストブロック・ブロックにまとめる → 読み順を付ける」でブロック矩形一覧を計算。
4. 得られた矩形を現在のキャンバス解像度に合わせてスケーリングし、選択状態として UI に反映。  
   レイアウト検出を実行した場合は、その結果をレイアウトキャッシュに保存（矩形一覧は都度計算し、キャッシュには検出結果だけ保存）。

### 6.2 OCR テキストレイヤー

1. ページのキャンバスが描画されると、そのページを OCR キューに載せる。
2. すでに OCR キャッシュがある場合
   - もともとテキスト埋め込みがあるページと判定されていれば、OCR は行わずスキップ。
   - 行キャッシュがあり、画像サイズが同じならそのまま表示。
   - 画像サイズが違う場合は、保存時のサイズと現在のサイズの比で bbox をスケーリングして表示（再 OCR はしない）。
3. OCR キャッシュがない場合
   - PDF にテキストが埋め込まれていれば OCR はスキップし、その旨を記録。
   - レイアウトキャッシュがあり、かつ画像解像度が今のキャンバスと一致していれば、「レイアウト検出はスキップし、行の切り出し＋読み順＋文字認識だけ」を Worker で実行。
   - レイアウトキャッシュがない、または解像度が違う場合は、レイアウト検出から行い、その結果をレイアウトキャッシュにも保存しつつ、行テキストを OCR キャッシュと UI に反映。

どちらのシーンでも、「レイアウト検出結果」は 1 回だけ計算してキャッシュに保存し、半自動矩形選択と OCR の両方で再利用できるようにしています。

## 7. 設定・モデル・環境

### 7.1 モデルと設定ファイル

- **ONNX モデル**: クライアントの公開ディレクトリ内の `ndlocr-lite/model/` に配置。
  - レイアウト検出: DEIM 用の ONNX ファイル。
  - 文字認識: PARSEQ 用の ONNX ファイル。
- **設定ファイル**: `ndlocr-lite/config/` に配置。
  - レイアウト検出用: クラス名（テキストブロック・行・表など）の定義。
  - 文字認識用: 文字セット（charlist）の定義。

Worker はこれらを読み込んで推論に使います。

### 7.2 ONNX Runtime Web（CPU スレッド数・ヘッダー）

- Worker 内で、ONNX セッション作成前に **WASM のスレッド数** を設定している。  
  負荷を抑えるため「論理コア数の半分、最低 1・最大 4」のような制限をしている。
- マルチスレッドには **SharedArrayBuffer** が必要で、ブラウザでは **Cross-Origin Isolation** が必須。
  - 開発サーバーでは、`Cross-Origin-Opener-Policy` と `Cross-Origin-Embedder-Policy` を付与する設定が必要。
  - 本番（Nginx 等）でも同様のヘッダーが必要。`require-corp` が厳しい場合は `credentialless` への変更が検討可能。

## 8. 座標とキャッシュの考え方

- Worker には「PDF ページを描画したキャンバスと同じ解像度の画像」を渡す想定。検出結果の座標はそのキャンバス座標系と一致する。
- **レイアウトキャッシュ**には、検出結果（領域一覧）と **画像の幅・高さ** を保存。
  - 半自動矩形選択: キャッシュ時の画像サイズと現在の表示サイズの比で矩形をスケーリングするため、解像度が変わってもキャッシュを再利用可能。
  - OCR: レイアウトキャッシュの画像サイズが現在のキャンバスと一致するときだけ「レイアウト検出スキップ」の経路を使い、違う場合はレイアウト検出からやり直す。
- **OCR キャッシュ**には、行の位置（bbox）とテキストに加え、OCR 実行時の画像サイズを保存。再表示時は、現在のキャンバス解像度との比で bbox をスケーリングして表示する。

## 9. 今後の拡張（プラン上のメモ）

- **ユーザー矩形のマージ**: 自動選択で、ユーザーが手動で描いた矩形もブロックとして扱い、検出結果とマージして読み順を付け直すなどの仕様が検討されている。
- **座標の一貫性**: レイアウト／OCR キャッシュとも画像サイズを持ち、表示時にスケーリングしている。将来的には手動矩形も含め「元画像座標」「表示座標」を整理し、ズーム・回転にも対応しやすくする。
- **PARSEQ のキャスケード**: 現状は 1 モデル（例: 100 文字用）のみ。必要に応じて文字数に応じたモデル振り分けを検討可能。
- **エラー・ローディング**: Worker や ONNX の失敗時はツールバーで「利用不可」「再試行」を表示し、OCR 中は該当ページにローディングを出すと UX が良い。

## 10. まとめ

- **レイアウトキャッシュ**: レイアウト検出（DEIM）の結果を保存。半自動矩形選択と OCR の両方で参照。ブロック矩形一覧は保存せず、必要なときにキャッシュから都度計算する。
- **半自動矩形選択**: ボタン押下でそのページのレイアウトを取得。キャッシュがあればそれを使ってブロック矩形を計算、なければレイアウト検出を実行してから矩形を計算し、結果をキャッシュに保存。得られた矩形を選択状態として表示する。
- **OCR テキストレイヤー**: レイアウトキャッシュがあり解像度が一致していれば文字認識（PARSEQ）のみ実行。そうでなければレイアウト検出から実行し、結果をレイアウトキャッシュにも保存しつつ、行テキストを OCR キャッシュと画面に反映。PDF にテキストが埋め込まれているページは OCR をスキップする。
- **キュー**: キューの仕組みを利用してメインスレッドと Worker で処理を負荷分散。表示中のページを優先することを前提としている。
- **CPU スレッド数**: ONNX Runtime Web のWASM版を利用してAIモデルをマルチスレッドCPUで処理をする。

以上の内容が、現在の実装とプランに基づく「AI 機能について」の整理です。  
より細かい実装位置（ファイル名・関数名）が必要な場合は、コード内のコメントや別の実装調査用ドキュメントを参照してください。
